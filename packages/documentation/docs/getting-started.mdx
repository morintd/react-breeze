# Getting Started

react-breeze is a component library working with configurable colors.

It allows for quick prototyping, while giving a good degree of freedom.

import { Button, Title } from 'react-breeze';

<Title as="h2">Installation</Title>

**yarn**

```shell
yarn add react-breeze
```

**npm**

```shell
npm add --save react-breeze
```

<Title as="h2">Configuration</Title>

Before using react-breeze, you need to [setup Tailwind CSS](https://tailwindcss.com/docs/installation).

<Title as="h3">Step 1: allow active variant</Title>

Next, you need to allow the variant _active_ for background color:

_tailwind.config.js_

```js
module.exports = {
  content: ['./src/**/*.{html,js}'],
  theme: {
    extend: {},
  },
  variants: {
    extend: {
      backgroundColor: ['active'],
    },
  },
  plugins: [],
};
```

<Title as="h3">Step 2: colors and safe list</Title>

Components in react-breeze work by passing them a color ([default color palette](https://tailwindcss.com/docs/customizing-colors)):

```jsx
import { Button } from 'react-breeze';

<Button color="blue">Button (blue)</Button>
<Button color="red">Button (red)</Button>
```

<Button className="mb-2 mr-1" color="blue">
  Button (blue)
</Button>
<Button className="mb-2 mr-1" color="red">
  Button (red)
</Button>

Classnames using those colors are generated dynamically, which means they will be removed from the CSS at build time. In order to work, react-breeze makes use of the [safelist](https://tailwindcss.com/docs/content-configuration#safelisting-classes) property inside `tailwind.config.js`.

React-breeze comes with an utility function `generateSafelist`. It generates all the classnames that shouldn't be removed, and simply needs the list of color you plan to use:

```js
const { generateSafelist } = require('react-breeze');

module.exports = {
  content: ['./src/**/*.{html,js}'],
  theme: {
    extend: {},
  },
  variants: {
    extend: {
      backgroundColor: ['active'],
    },
  },
  plugins: [],
  safelist: generateSafelist(['blue', 'red']),
};
```

<Title as="h3">Step 3: add inverted variant</Title>

Some components need an `inverted` variant, used for example in the button text color. In the example from _step 2_, this variant has been set to white (`#ffffff`).

You have to set this variant for every used color:

```js
module.exports = {
  content: ['./src/**/*.{html,js}'],
  theme: {
    extend: {
      colors: {
        red: {
          inverted: '#ffffff',
        },
        blue: {
          inverted: '#ffffff',
        },
      },
    },
  },
  variants: {
    extend: {
      backgroundColor: ['active'],
    },
  },
  plugins: [],
  safelist: generateSafelist(['blue', 'red']),
};
```

That's it, you're good to go! <a href="/docs/components/alert">Explore components</a>

<Title as="h4">Design system</Title>

Using **raw colors** (`blue`, `red`, ..) everywhere, will make your app more prone to errors, and harder to maintain as time goes on.

Instead, you can use colors that show **intent**. For example, you can build a design system based on:

<Title as="h4">Color definition</Title>

Instead of passing a color string to components (`color="blue"`, `color="red"`), which is also error-prone, you can define an object that stores the available colors.

With TypeScript, an enum does an amazing job:

```tsx
enum Color {
  Primary = 'primary',
  Secondary = 'secondary',
  Success = 'success',
  Info = 'info',
  Warning = 'warning',
  Danger = 'danger',
}

<Button color={Color.Primary}>Button</Button>;
```

<Button className="mb-2 mr-1" color="blue">
  Button
</Button>

_(Feel free to change those colors.)_
